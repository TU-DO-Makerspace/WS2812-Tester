116  /**
117  * @brief Returns if the encoder is considered "ready" to be read
118  *
119  * The following function returns if the rotary encoder is considered
120  * "ready". To ensure the firmware can keep up with changes in the rotary
121  * encoder, we must ensure that the task of reading the rotary encoder
122  * is provided a high priority where no other tasks can interrupt the
123  * firmware from registering changes.
124  * Typically hardware interrupts can solve this issue, however, since
125  * writing to the LED strip requires hardware interrupts to be disabled,
126  * just relying in interrupts in itself will still result in rotary encoder
127  * changes getting "skipped" (especially as the LED strip size increases as
128  * transmissions take longer to reach the end of the strip).
129  * Thus, the solution to fix this issue is to halt the firmware from doing
130  * anything else than listening for rotary encoder change until the rotary
131  * encoder has remained unchaged for a specific ammount of time (See ready_time_ms),
132  * after which it is considered "ready" to be read.
133  * The drawback of this is that the firmware cannot do anything else in the meantime,
134  * which for our application however, is perfectly acceptable.
135  *
136  * @returns If the encoder is considered "ready" to be read
137  */
138  bool ready();
139 
140  /**

144  * high priority, meaning running time consuming tasks should be avoided when calling

146  * will result in sluggish/laggy reading of the encoder.
147  *
148  * @returns If rotary encoder position has changed
149  */
